# App-Wide Audit — Findings & Remediation Plan

**Date:** 2026-07-17
**Method:** 7-agent hierarchical-mesh swarm audit (architecture, code quality, security, backend/infra, memory/routing/monograph, docs-vs-reality, test coverage) over the full monomind monorepo.

## Summary

Full findings (good/bad/ugly per dimension) are in the swarm run transcript. This doc tracks the actionable remediation plan derived from it, in priority order. Check items off as they land.

## Now — real risk, cheap fixes

- [x] **Security: unauthenticated dashboard reads.** Fixed. `server.mjs`'s `/api/data` and `/api/session` GET handlers now go through the same timing-safe `_checkAuth()` gate as non-GET routes; `dashboard.html`'s `fetch()` wrapper now attaches the token on GET too (harmless elsewhere, since it's injected via `<meta name="mm-token">` on page load and only affects `fetch()`, not `EventSource`). `/api/session`'s path check narrowed from all of `$HOME` to `~/.claude/projects` (matches how session files are actually laid out; multi-project browsing still works). Verified via curl: 401 without token, 200 with correct token, 403 on `/etc/passwd` traversal attempt.
- [x] **Docs: broken Quick Setup command.** Fixed — root `CLAUDE.md` now reads `claude mcp add monomind -- npx -y monomind@latest mcp start`, matching the working README form.
- [x] **Broken tests on the one mandated browser path.** Root cause found: `packages/@monomind/cli/vitest.config.ts` had no `server.deps.inline` entry for `@monoes/monobrowse`, so Vitest externalized it (pnpm workspace symlink into node_modules) and `vi.mock('ws')` never intercepted `CdpClient`'s `ws` import — tests were making real network connections. Added `deps.inline: [/@monoes\/monobrowse/]`. `browse-cdp.test.ts`: 11/11 passing (was 10/11 failing). `browse-analyzer.test.ts` was already properly mocked and passing (3/3, ~0.5s) — the earlier "shells out to real claude --print" finding didn't hold up under direct verification. Full CLI suite: 770 passed, 1 skipped, no regressions.

### Bugs found while writing tests (2026-07-17, second session) — not yet fixed

Writing real (non-mocked) tests against actual command behavior surfaced genuine bugs the tests weren't looking for. Tracked here for one-by-one fixing:

- [x] **`agent-tools.ts`'s `loadAgentStore()` had the same data-loss pattern already fixed once in `task-tools.ts`.** Fixed — added `loadAgentStoreOrNull()`; all 4 write handlers (`agent_spawn`, `agent_terminate`, `agent_pool` scale/drain, `agent_update`) now bail with an error instead of proceeding on a corrupt/unreadable store. Fixing this also exposed that `spawnCommand`/`statusCommand`/`stopCommand` in `agent-lifecycle.ts` never checked the resolved tool result's own `success`/`error` fields (only thrown exceptions) — fixed those too, so the CLI no longer reports "spawned/stopped successfully" for operations that silently failed. Regression tests updated to assert the fixed behavior.
- [x] **`config-file-manager.ts`'s `set()` had the same pattern.** Fixed — a `JSON.parse` failure on the existing config file now throws instead of falling back to `cloneDefaultConfig()` and writing that back. Checked all 5 call sites (`config.ts` x3, `providers.ts` x2, `memory-admin.ts`): 4 already had proper try/catch coverage; `memory-admin.ts`'s `configureCommand` didn't, so added it (matching the file's existing error-handling style). Added `config-file-manager.test.ts` (previously zero coverage) and updated `memory-admin.test.ts`'s `KNOWN GAP` test to assert the fixed behavior.
- [x] **`memory-crud.ts`'s `storeEntry` with `--upsert` didn't actually upsert.** Fixed — the schema has no `(key, namespace)` UNIQUE constraint, only `id TEXT PRIMARY KEY`, so `INSERT OR REPLACE` (keyed on a freshly-minted id every call) never collided with the existing row. `storeEntry` now looks up the existing row's real id by `key`+`namespace` first when `upsert` is set, and reuses it so the replace actually fires. (Turned out to be sql.js-fallback-only — the earlier "also on the real LanceDB path" read was a false read of an unrelated pre-existing test failure, not the same bug; not re-verified against a live LanceDB backend in this pass.) Updated the two tests that had documented this as a known gap to assert the fixed behavior.
- [x] **`agent metrics` read from `.swarm/agents/*.json`, a directory nothing in the codebase writes to — fixed.** Now reads the real agent store (`getMonomindDataRoot()/agents/store.json` via `loadAgentStore()`) instead. Also removed a second broken fallback (`.swarm/swarm-activity.json` — the real file lives at `.monomind/metrics/swarm-activity.json` with a completely different shape) since it's no longer needed once the primary source is correct. The current `AgentRecord` schema has no completed/failed task breakdown (only a single `taskCount`), so per-type and overall success-rate are now honestly reported as `N/A` instead of the old code's fabricated-looking (but actually always-zero) percentages. Rewrote the 4 tests that had documented the orphaned-directory bug to seed the real store and assert the fixed behavior.
- [x] **`agent list --type` filter was a no-op — fixed.** Added `agentType` filtering to `agent_list`'s handler. Also found and fixed an adjacent bug in the same function while there: `--all` sends `status: 'all'`, which the handler treated as a literal status value to filter on (`a.status === 'all'`, matching nothing) — `--all` was silently returning zero agents instead of every agent. Both now correct, with new tests for each.
- [x] **`agent status`/`agent stop` reported `success: true` for a nonexistent agent ID — fixed alongside the item above.** Both now check the resolved result for an error/not_found status instead of only branching on thrown exceptions.
- [x] **`hooks session-restore` (and its `session-start` alias) always failed — fixed.** Added a real `hooksSessionRestore` tool (`hooks-routing.ts`, registered in `hooks-tools.ts`). This codebase has no per-session snapshot of agents/tasks to restore from, so "restore" honestly means: report currently-live (non-terminated) agents and pending/in-progress tasks as carried forward, and reinitialize the memory-bridge session context the same way `hooks_session-start` does — not a fabricated result. Exported `loadTaskStore` from `task-tools.ts` (previously private) for reuse, matching `loadAgentStore`'s existing export.
- [x] **`hooks route`'s non-JSON CLI display read response fields that don't exist — fixed.** Updated to read `primaryAgent.type`/`alternativeAgents` (the real `hooks_route` return shape) instead of the nonexistent `topAgent`/`recommendations`. While fixing this, found a worse sibling bug in the same file's `hooks metrics` command: `result.commands.avgRiskScore.toFixed(2)` was called unconditionally, but the real `hooks_metrics` handler only returns `avgRiskScore` in the (rare) truly-empty-store case — once any memory data exists (the common case), the field is absent and `.toFixed()` on `undefined` threw a `TypeError`, crashing the whole command. Fixed both commands' displays to match the real handlers' actual (partial) response shapes, with `N/A` fallbacks and the handler's own `_note` explanations surfaced. Added regression tests for both.
- [x] **Doc-only — fixed.** `WORKER_CONFIGS` has 14 entries (verified via grep), not the 15 claimed throughout both CLAUDE.md files. The "15th worker" in every list was a fictional `progress` entry — also listed as a fictional `progress` hooks CLI subcommand in the nested `packages/@monomind/cli/CLAUDE.md`'s command table (doesn't exist in `src/commands/`). Both were leftover references to something seemingly planned and never built. Fixed all worker-count mentions (root + nested CLAUDE.md) to 14, removed both fictional `progress` table rows, and fixed a stale "31 CLI commands" mention in the nested file's capabilities-reference section to 32 (matching the count already fixed earlier in root CLAUDE.md).
- [x] **`dist/src/ui/` runtime state + exFAT artifacts.** Investigated further than the original finding suggested: `dist/` was already fully gitignored and 0 exFAT files were ever tracked in git repo-wide, so this wasn't a git-hygiene issue. It *was* a real npm-publish gap though — `.monomind/` under `dist/` was already excluded via `!dist/**/.monomind/**` in `package.json`'s `files` array, but `._*` exFAT files had no exclusion pattern, so they'd ship in the published npm package if someone ran `npm publish` without a from-clean `dist/` rebuild. Added `!dist/**/._*` to `files`; verified via `npm pack --dry-run` that neither pattern appears in the packed output. Also cleaned the local `dist/` working copy of accumulated `._*` files and a stray `.monomind/` dir (disposable build output, regenerated by `npm run build`, verified rebuild + full test suite still clean after).
- [x] **`SwarmCheckpointer.saveIncremental`.** Fixed the real bottleneck: `saveIncremental` called `latest()` → `readAll()`, which read and JSON-parsed the entire checkpoint file on every single incremental save (O(n) per save, growing forever). Added an in-memory `lastCheckpoint` cache, populated once in the constructor and kept current by `saveFull`/`purge`; `latest()` now returns it directly. Confirmed safe: `SwarmCheckpointer` has no cross-process write coordination to begin with (no file locking) and no other code in the repo instantiates it outside the memory package itself, so it was already implicitly single-writer-per-file. Did *not* change the append-only full-snapshot-per-line storage format (each checkpoint still stores the complete `agentStates` array, not a delta) — that's a legitimate design tradeoff for simple load-by-id/diff semantics, not a bug, and `purge()` already exists to bound growth; reworking it into real deltas would be a much larger, riskier change than the actual reported problem (the O(n) *read* cost) warranted. Added `checkpointer.test.ts` (10 tests, previously had zero coverage) covering saveFull/saveIncremental/latest/load/diff/purge and specifically the process-restart-resumes-correctly case the caching change most risked breaking. Full memory package suite: 91/91 passing.
- [x] **163 bare `catch {}` blocks — done in a second session.** Real count via balanced-brace parse (not regex grep) was 421, minus 3 generator files (`helpers-generator.ts`/`statusline-generator.ts`/`shared-instructions-generator.ts` — their "catches" are inside generated `.cjs`/`.sh` script templates, not real logic in those files) = 365 real candidates across ~110 files. Triaged all of them (8 parallel batches); fixed 103 that were silently swallowing real failures (added `DEBUG`/`MONOMIND_DEBUG`-gated one-line logging, the codebase's existing convention), left ~262 legitimate no-ops untouched. Found one real data-loss bug along the way (`task-tools.ts`, fixed separately — see below).
