# monobrowse Workflow System — Design Spec

**Date:** 2026-06-20  
**Status:** Approved

---

## Overview

A browser workflow automation system baked into `monomind browse`. Users define DAG-based workflows in JSON, run them via CLI, and monitor execution in a persistent web dashboard. An AI-powered action builder generates reusable action definitions by inspecting any website's DOM.

---

## Decisions

| Decision | Choice |
|---|---|
| Dashboard | Web browser tab (local server, persists between runs) |
| Action builder | AI Page Analyzer (Claude inspects DOM → generates step JSON) |
| Platforms | LinkedIn, Instagram, X, Gemini |
| Storage | Hybrid — JSON files (workflows/actions) + SQLite (run history, sessions) |
| Architecture | New files under `browse` namespace, `browse.ts` untouched |

---

## Command Structure

All new commands extend `monomind browse` as new subcommand groups.

### `browse workflow`

```
monomind browse workflow create <name>       # scaffold workflow JSON interactively
monomind browse workflow run <file.json>     # execute; auto-opens web dashboard
monomind browse workflow list                # list workflows + last run status
monomind browse workflow status <run-id>     # check a specific execution
monomind browse workflow stop <run-id>       # cancel a running workflow
```

### `browse action`

```
monomind browse action build --url <url> --task "<description>"   # AI analyzer → generates action JSON
monomind browse action run <action-id> --account <user> [--params k=v ...]
monomind browse action list                  # built-in + custom actions
monomind browse action show <action-id>      # print action definition
```

### `browse platform`

```
monomind browse platform connect <platform>  # open browser → login → save session
monomind browse platform list                # show connected accounts + session status
monomind browse platform disconnect <id>     # remove saved session
```

Supported platforms: `linkedin`, `instagram`, `x`, `gemini`

## Data Formats

### Workflow Definition (`.monomind/workflows/<name>.json`)

Git-committable, human-editable. Source of truth for workflow logic.

```json
{
  "id": "linkedin-outreach",
  "name": "LinkedIn Outreach",
  "nodes": [
    { "id": "trigger", "type": "trigger.manual", "config": {} },
    {
      "id": "comment",
      "type": "action.linkedin.comment_post",
      "config": {
        "account": "{{$env.LINKEDIN_USER}}",
        "post_url": "{{$json.url}}",
        "text": "{{$json.comment}}"
      }
    }
  ],
  "connections": [
    { "from": "trigger", "to": "comment" }
  ]
}
```

**Node types (v1):**
- `trigger.manual` — one-shot, started by `workflow run`
- `action.<platform>.<action_id>` — runs a platform action
- `core.if` — conditional branch (`true`/`false` outputs)
- `core.set` — transform item fields
- `core.filter` — predicate filter

### Action Definition (`.monomind/actions/<id>.json`)

Generated by `action build` or written by hand. Shipped built-in actions live in `browser/adapters/`.

```json
{
  "id": "linkedin:comment_post",
  "platform": "linkedin",
  "name": "Comment on Post",
  "params": ["post_url", "text"],
  "steps": [
    { "type": "navigate", "url": "{{params.post_url}}" },
    { "type": "find", "selectors": [".comment-box", "[aria-label='Add a comment']"], "as": "box" },
    { "type": "click", "target": "{{box}}" },
    { "type": "type", "target": "{{box}}", "text": "{{params.text}}", "humanDelay": true },
    { "type": "find", "selectors": ["button[type=submit]", ".comment-submit"], "as": "submit" },
    { "type": "click", "target": "{{submit}}" },
    { "type": "wait", "condition": "network_idle", "timeout": 3000 }
  ]
}
```

**Step types:** `navigate`, `find` (multi-selector fallback), `click`, `type`, `wait`, `extract`, `condition`

**`find` fallback:** Tries each selector in order, uses first that resolves. Generates up to 3 selectors per element (CSS, aria-label, XPath) to survive platform UI changes.

## Module Responsibilities

### `workflow/engine.ts`

- Loads `WorkflowDef` from store
- Builds DAG via topological sort (Kahn's algorithm), rejects cycles
- Executes nodes in dependency order; routes `Item[]` outputs via connections
- Emits events to dashboard server on each step start/complete/fail
- Handles merge nodes (waits for all predecessors via in-degree counter)
- Cancellable via `AbortSignal`

### `workflow/store.ts`

- `readWorkflow(path)` — parses JSON file, validates schema
- `writeRunRecord(record)` — inserts into SQLite `browse_runs` table
- `listRuns(workflowId?)` — queries run history
- SQLite path: `~/.monomind/browse.db`

### `workflow/expression.ts`

- Resolves `{{$json.field}}` from current item
- Resolves `{{$env.VAR}}` from `process.env`
- Resolves `{{params.name}}` from action invocation params
- Resolves `{{$node["NodeId"].field}}` from prior node output
- Caches compiled templates with `Map<string, CompiledTemplate>`

### `adapters/index.ts`

```typescript
interface PlatformAdapter {
  platform: string;                                    // "linkedin"
  baseURL: string;                                     // "https://www.linkedin.com"
  reservedPaths: string[];                             // ["/feed", "/jobs", "/messaging"]
  isLoggedIn(page: PageInterface): Promise<boolean>;
  loginURL(): string;
  extractUsername(page: PageInterface): Promise<string>;
}
```

Adapters handle login detection only. Action steps come from action definition files.

### `action-builder/analyzer.ts`

1. Connect to browser via existing `CdpClient`
2. Navigate to `--url`
3. Capture: accessibility tree snapshot + `document.title` + visible interactive elements (inputs, buttons, links)
4. Send DOM snapshot + `--task` description to Claude API (`claude-sonnet-4-6`)
5. Prompt instructs Claude to output an `ActionDef` JSON with multi-selector `find` steps
6. Validate response schema, write to `.monomind/actions/<id>.json`
7. Print generated file path + preview

### `dashboard/server.ts`

- Starts HTTP server on port 4242 (configurable via `MONOBROWSE_DASHBOARD_PORT`)
- Serves `dashboard/ui.html` at `/`
- WebSocket endpoint at `/ws` — broadcasts `RunEvent` messages from engine
- `RunEvent` types: `run_started`, `step_started`, `step_completed`, `step_failed`, `run_completed`, `run_stopped`
- Keeps last 50 run records in memory for page refresh
- Started lazily on first `workflow run`; stays alive until CLI process exits

### `dashboard/ui.html`

- Self-contained HTML file bundled with the CLI (no external dependencies)
- Connects to `ws://localhost:4242/ws` on load
- Shows active workflow cards with step list, item progress bar, elapsed time, log tail
- Completed runs remain visible (dimmed)
- Stop button per running workflow (sends `POST /stop/:runId`)

## Session / Credential Storage

Sessions stored in SQLite `browse_sessions` table:
```
id TEXT PRIMARY KEY    -- "linkedin:john@example.com"
platform TEXT
username TEXT
cookies TEXT           -- JSON serialized
user_agent TEXT
created_at INTEGER
last_used_at INTEGER
```

`platform connect` opens a browser window to the platform login URL, waits for successful login detection (`adapter.isLoggedIn()`), extracts username, serializes cookies, stores in DB.

Action execution resolves session by `platform + account` from node config.

---

## Expression Engine

Templates use `{{double-brace}}` syntax resolved before each step:

| Expression | Resolves to |
|---|---|
| `{{$json.url}}` | Field from current item |
| `{{$env.LINKEDIN_USER}}` | Environment variable |
| `{{params.text}}` | Action invocation parameter |
| `{{$node["comment"].result}}` | Output of prior node |
| `{{box}}` | Named element ref from prior `find` step |

Unresolved expressions throw at execution time (fail-fast, surface config errors early).

---

## Error Handling

- **Step failure:** Log error + item, continue with next item (configurable: `onError: "skip" | "stop"` per node)
- **Login not detected:** `platform connect` prompts user to complete login manually, polls `isLoggedIn()` every 2s with 60s timeout
- **Selector not found:** `find` step tries each selector in order; throws `SelectorNotFoundError` after all fail
- **Dashboard unreachable:** Engine continues; emits warning, does not block execution
- **Claude API failure in `action build`:** Surfaces error with raw DOM excerpt so user can debug prompt

## Storage Layout

```
.monomind/
  workflows/          ← workflow JSON files (project-local, git-committable)
  actions/            ← custom action JSON files (generated by action build)

~/.monomind/
  browse.db           ← SQLite: run history, sessions, credentials
    tables: browse_runs, browse_sessions
```

## Out of Scope (v1)

- Scheduled triggers (`trigger.schedule`) — manual only in v1
- Workflow editor GUI — CLI + JSON files only
- Parallel node execution — sequential only in v1
- Webhook triggers
- Action marketplace / sharing
